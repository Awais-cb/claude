Skills are custom /commands you define with a markdown file. Instead of typing the same long set of instructions every time, define them once as a skill and trigger the whole workflow with a short command.

Without skills:
  "Review my code for security issues, check for hardcoded credentials,
   validate all SQL uses parameterized inputs, check auth on protected routes..."

With skills:
  /security-check

Claude also auto-invokes skills when relevant to your prompt — you don't always have to type the command yourself. Skills follow the Agent Skills open standard and work across multiple AI tools.

Bundled Skills#

These ship with Claude Code and are available in every session:

CommandWhat it does
/simplifySpawns three parallel review agents (code reuse, quality, efficiency), aggregates findings, applies fixes. Run after implementing a feature.
/batch <instruction>Orchestrates large-scale changes in parallel. Decomposes the work into 5–30 units, shows a plan for approval, then spawns one agent per unit in isolated git worktrees.
/debug [description]Reads the session debug log to troubleshoot your current Claude Code session.

Where Skills Live#

When skills share the same name across levels, priority order is: Enterprise > Personal > Project. Plugin skills use a plugin-name:skill-name namespace so they never conflict.

Each skill is a directory with SKILL.md as the entrypoint:

my-skill/
├── SKILL.md           ← required
├── template.md        ← optional supporting files
├── examples/
│   └── sample.md
└── scripts/
    └── validate.sh

Create Your First Skill#

# Personal skill — available in all your projects
mkdir -p ~/.claude/skills/explain-code

Create ~/.claude/skills/explain-code/SKILL.md:

---
name: explain-code
description: Explains code with visual diagrams and analogies. Use when explaining how code works or when the user asks "how does this work?"
---

When explaining code, always include:

1. **Start with an analogy** — compare to something from everyday life
2. **Draw a diagram** — use ASCII art to show the flow or structure
3. **Walk through step-by-step** — explain what happens in sequence
4. **Highlight a gotcha** — common mistake or misconception to avoid

Test it:

# Auto-invoked when relevant:
How does the auth middleware work?

# Or invoke directly:
/explain-code src/auth/middleware.ts

Frontmatter Reference#

FieldRequiredDescription
nameNoSkill name (defaults to directory name). Lowercase, numbers, hyphens only, max 64 chars.
descriptionRecommendedWhat the skill does and when to use it. Claude uses this to decide when to auto-invoke.
argument-hintNoHint shown in autocomplete, e.g. [issue-number]
disable-model-invocationNoSet true to prevent Claude auto-loading this skill. Only you can invoke it with /name.
user-invocableNoSet false to hide from the / menu. Only Claude can invoke it.
allowed-toolsNoTools Claude can use without asking permission when this skill is active.
modelNoModel to use when this skill is active.
contextNoSet fork to run in an isolated subagent context.
agentNoWhich subagent type to use with context: fork (e.g. Explore, Plan).

Controlling Who Invokes a Skill#

FrontmatterYou can invokeClaude can invokeWhen loaded
(default)YesYesDescription in context; full skill loads on invoke
disable-model-invocation: trueYesNoNot in context; loads when you invoke
user-invocable: falseNoYesDescription in context; loads when Claude invokes

Use disable-model-invocation: true for skills with side effects you want to control timing on — like /deploy or /send-slack-message. You don't want Claude deciding to deploy because the code looks ready.

---
name: deploy
description: Deploy the application to production
disable-model-invocation: true
---

Deploy $ARGUMENTS to production:
1. Run the test suite
2. Build the application
3. Push to deployment target
4. Verify deployment succeeded

Passing Arguments#

Arguments passed after the skill name are available via $ARGUMENTS:

---
name: fix-issue
description: Fix a GitHub issue by number
---

Fix GitHub issue $ARGUMENTS following our coding standards.

/fix-issue 123 → Claude receives "Fix GitHub issue 123 following our coding standards."

Access individual arguments by position with $ARGUMENTS[N] or the shorthand $N:

---
name: migrate-component
description: Migrate a component between frameworks
---

Migrate the $0 component from $1 to $2.
Preserve all existing behaviour and tests.

/migrate-component SearchBar React Vue → replaces $0 with SearchBar, $1 with React, $2 with Vue.

Dynamic Context Injection (!`command`)#

The !`command` syntax runs a shell command before the skill is sent to Claude. The output replaces the placeholder in the prompt — Claude receives the rendered result, not the command itself.

---
name: pr-summary
description: Summarize the current pull request
context: fork
agent: Explore
allowed-tools: Bash(gh *)
---

## Pull request context
- Diff: !`gh pr diff`
- Comments: !`gh pr view --comments`
- Changed files: !`gh pr diff --name-only`

## Task
Summarize this pull request concisely.

When this skill runs, each !`...` executes first and the output is substituted in. Claude only sees the final rendered prompt with real PR data.

Running in a Subagent (context: fork)#

Add context: fork to run the skill in isolation — it won't have access to your conversation history. The skill content becomes the subagent's task. Results are summarised and returned to your main conversation.

---
name: deep-research
description: Research a topic thoroughly in the codebase
context: fork
agent: Explore
---

Research $ARGUMENTS thoroughly:

1. Find relevant files using Glob and Grep
2. Read and analyse the code
3. Summarise findings with specific file references

The agent field specifies which subagent type to use. Options: built-in types (Explore, Plan, general-purpose) or any custom subagent from .claude/agents/. Omit to default to general-purpose.

Restricting Tool Access#

Use allowed-tools to limit which tools Claude can use when a skill is active:

---
name: safe-reader
description: Read files without making any changes
allowed-tools: Read, Grep, Glob
---

Read and analyse files only. Do not edit anything.

Legacy: .claude/commands/#

The older .claude/commands/ directory still works — a file at .claude/commands/review.md creates a /review command exactly like a skill would. Skills are the recommended approach since they support additional features: supporting files, frontmatter invocation control, and context: fork. If both exist with the same name, the skill takes precedence.

Troubleshooting#

Skill not triggering: Check the description includes keywords users would naturally say. Try invoking directly with /skill-name to verify it loads.

Skill triggers too often: Make the description more specific, or add disable-model-invocation: true to require manual invocation.

Claude doesn't see all skills: Skill descriptions load into context up to a 2% context window budget (min 16,000 chars). Run /context to check for a "skills excluded" warning. Override the limit with the SLASH_COMMAND_TOOL_CHAR_BUDGET environment variable.
