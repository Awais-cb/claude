Plugins let you package skills, agents, hooks, and MCP servers into a single shareable unit. Instead of manually copying .claude/ files across projects or teammates' machines, you create a plugin once and install it anywhere.

Standalone (.claude/)         Plugin
─────────────────────         ──────────────────────────
Available in one project  →   Shared via marketplaces
Files copied manually     →   Installed with /plugin
Short names: /hello       →   Namespaced: /my-plugin:hello
No version tracking       →   Semantic versioning

Plugins vs Standalone Config#

Use standalone .claude/ when…Use a plugin when…
Customising a single projectSharing with your team or community
Personal config, doesn't need sharingSame skills/agents needed across multiple projects
Quick experimentsYou want version control and easy updates
You want short names like /helloDistributing through a marketplace

Tip: Start with .claude/ for quick iteration, then convert to a plugin when you're ready to share.

Plugin Structure#

my-plugin/
├── .claude-plugin/
│   └── plugin.json         ← manifest (required)
├── skills/
│   └── code-review/
│       └── SKILL.md        ← creates /my-plugin:code-review
├── agents/
│   └── reviewer.md         ← custom subagent
├── hooks/
│   └── hooks.json          ← event hooks
├── .mcp.json               ← bundled MCP servers
├── settings.json           ← default settings
└── README.md               ← user documentation

Common mistake: Only plugin.json goes inside .claude-plugin/. All other directories (skills/, agents/, hooks/) must be at the plugin root level.

Create a Plugin#

1. Create the directory and manifest

mkdir -p my-plugin/.claude-plugin

Create my-plugin/.claude-plugin/plugin.json:

{
  "name": "my-plugin",
  "description": "My custom Claude Code extensions",
  "version": "1.0.0",
  "author": {
    "name": "Your Name"
  }
}
FieldPurpose
nameUnique identifier and skill namespace. Skills become /my-plugin:skill-name.
descriptionShown in the plugin manager.
versionSemantic versioning for releases.
authorOptional attribution.

2. Add a skill

mkdir -p my-plugin/skills/hello

Create my-plugin/skills/hello/SKILL.md:

---
description: Greet the user with a friendly message
---

Greet the user named "$ARGUMENTS" warmly and ask how you can help.

3. Test locally

claude --plugin-dir ./my-plugin
/my-plugin:hello Alex

Load multiple plugins at once:

claude --plugin-dir ./plugin-one --plugin-dir ./plugin-two

Skill Namespacing#

Plugin skills are always namespaced to prevent conflicts. A hello skill in a plugin named my-plugin becomes /my-plugin:hello. To change the prefix, update the name field in plugin.json.

Claude's auto-invocation still works — if the skill description matches your prompt, Claude loads it without you typing the full command.

Add Hooks to a Plugin#

Create my-plugin/hooks/hooks.json with the same format as settings.json hooks:

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npm run lint:fix"
          }
        ]
      }
    ]
  }
}

Bundle MCP Servers#

Create my-plugin/.mcp.json to include MCP servers that start automatically when the plugin is enabled:

{
  "mcpServers": {
    "database-tools": {
      "command": "${CLAUDE_PLUGIN_ROOT}/servers/db-server",
      "args": ["--config", "${CLAUDE_PLUGIN_ROOT}/config.json"],
      "env": {
        "DB_URL": "${DB_URL}"
      }
    }
  }
}

Use ${CLAUDE_PLUGIN_ROOT} for paths relative to the plugin directory. MCP server changes require restarting Claude Code to take effect.

Default Settings#

Create my-plugin/settings.json to apply configuration when the plugin is enabled. Currently supports the agent key to activate a custom agent as the main thread:

{
  "agent": "security-reviewer"
}

This activates the security-reviewer agent from the plugin's agents/ directory, changing how Claude Code behaves by default when the plugin is enabled.

Managing Plugins#

Use the /plugin command in Claude Code to manage installed plugins:

/plugin              # Open the plugin manager
/plugin install      # Install a plugin from a marketplace
/plugin list         # Show all installed plugins

Plugin skills appear in the / menu with their namespace prefix. Run /mcp to see MCP servers bundled by plugins.

Convert Existing Config to a Plugin#

Already have skills or hooks in .claude/? Migrate them:

# 1. Create plugin structure
mkdir -p my-plugin/.claude-plugin

# 2. Write the manifest
cat > my-plugin/.claude-plugin/plugin.json << 'EOF'
{
  "name": "my-plugin",
  "description": "Migrated from standalone configuration",
  "version": "1.0.0"
}
EOF

# 3. Copy existing files
cp -r .claude/skills my-plugin/     # if you have skills
cp -r .claude/commands my-plugin/   # if you have commands
cp -r .claude/agents my-plugin/     # if you have agents

# 4. Migrate hooks (copy the hooks object from settings.json)
mkdir my-plugin/hooks
# create my-plugin/hooks/hooks.json with your hooks

# 5. Test it
claude --plugin-dir ./my-plugin
StandalonePlugin equivalent
.claude/skills/my-plugin/skills/
.claude/commands/my-plugin/commands/
.claude/agents/my-plugin/agents/
Hooks in settings.jsonmy-plugin/hooks/hooks.json

Share & Distribute#

When your plugin is ready to share:

  1. Add a README.md with installation and usage instructions
  2. Version it using semantic versioning in plugin.json
  3. Publish to a marketplace — a marketplace is simply a directory or git repo listing plugins your team can install from /plugin install
  4. Submit to the official Anthropic marketplace at claude.ai/settings/plugins/submit

Troubleshooting#

Plugin not loading: Confirm directories are at the plugin root, not inside .claude-plugin/. Only plugin.json belongs in .claude-plugin/.

Skills not appearing: Check the skill has a valid SKILL.md with frontmatter. Restart Claude Code after making changes.

MCP servers not starting: MCP server changes require restarting Claude Code. Run /mcp to see if the server appears in the list.

Wrong skill name: Skill names are always prefixed with the plugin's name field from plugin.json. Check that field matches what you expect.
