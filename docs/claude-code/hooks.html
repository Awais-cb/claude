Hooks — Automate Workflows#

Hooks are shell commands (or HTTP calls, or LLM prompts) that run automatically at specific points in Claude Code's lifecycle. They give you deterministic control over Claude's behaviour — ensuring certain actions always happen rather than relying on Claude to choose to run them.

Common uses:
  Auto-format code after every file edit
  Desktop notification when Claude needs input
  Block edits to protected files like .env
  Re-inject context after compaction
  Audit every command Claude runs
  Verify tests pass before Claude stops

Your First Hook in 2 Minutes#

The fastest way to add a hook is via /hooks in Claude Code — an interactive menu that walks you through picking an event, setting a matcher, and entering your command. No JSON editing required.

Or add one directly to your settings file. This sends a desktop notification whenever Claude finishes and needs your input:

// ~/.claude/settings.json
{
  "hooks": {
    "Notification": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "osascript -e 'display notification \"Claude needs your attention\" with title \"Claude Code\"'"
          }
        ]
      }
    ]
  }
}

Linux: replace the command with notify-send 'Claude Code' 'Claude needs your attention'
Windows: use powershell.exe -Command "[System.Windows.Forms.MessageBox]::Show('Claude needs your attention')"

Hook Configuration Format#

Hooks live inside the "hooks" key of any settings.json file. Each event maps to an array of matcher groups, and each group contains an array of hooks to run:

{
  "hooks": {
    "PostToolUse": [          // event name
      {
        "matcher": "Edit|Write",  // filter: regex on tool name
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

All matching hooks within a group run in parallel. Identical commands are automatically deduplicated.

Matchers — Filtering When Hooks Fire#

Without a matcher, a hook fires on every occurrence of its event. A matcher is a regex pattern applied to a specific field depending on the event:

EventMatcher filters onExample values
PreToolUse, PostToolUse, PostToolUseFailure, PermissionRequestTool nameBash, Edit|Write, mcp__github__.*
SessionStartHow the session startedstartup, resume, clear, compact
SessionEndWhy the session endedclear, logout, prompt_input_exit
NotificationNotification typepermission_prompt, idle_prompt
SubagentStart, SubagentStopAgent typeExplore, Plan, custom agent names
PreCompactWhat triggered compactionmanual, auto
ConfigChangeConfig sourceuser_settings, project_settings, skills
UserPromptSubmit, Stop, TeammateIdle, TaskCompletedNo matcher supportAlways fires

MCP tools use the naming convention mcp__<server>__<tool>. Match all tools from one server with mcp__github__.*.

Input, Output & Exit Codes#

When a hook fires, Claude Code passes event data as JSON to your script's stdin. For example, a PreToolUse hook for Bash receives:

{
  "session_id": "abc123",
  "cwd": "/Users/you/myproject",
  "hook_event_name": "PreToolUse",
  "tool_name": "Bash",
  "tool_input": {
    "command": "npm test"
  }
}

Your script's exit code tells Claude Code what to do next:

Exit codeMeaningNotes
0Allow the action to proceedFor SessionStart and UserPromptSubmit, stdout is injected into Claude's context
2Block the actionWrite the reason to stderr — Claude receives it as feedback and adjusts
Any otherAction proceeds; hook output is logged onlyVisible with Ctrl+O verbose mode

For more control than allow/block, exit 0 and print a JSON object to stdout. For PreToolUse:

{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",
    "permissionDecisionReason": "Use rg instead of grep for better performance"
  }
}

permissionDecision options: "allow" (no prompt), "deny" (cancel + feedback), "ask" (show prompt to user).

Command Hooks — Practical Examples#

Auto-format after every edit#

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

Block edits to protected files#

Save as .claude/hooks/protect-files.sh (run chmod +x on it), then register:

#!/bin/bash
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
PROTECTED=(".env" "package-lock.json" ".git/")
for pattern in "${PROTECTED[@]}"; do
  if [[ "$FILE_PATH" == *"$pattern"* ]]; then
    echo "Blocked: $FILE_PATH matches protected pattern '$pattern'" >&2
    exit 2
  fi
done
exit 0
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/protect-files.sh"
          }
        ]
      }
    ]
  }
}

Re-inject context after compaction#

Anything your command writes to stdout is added to Claude's context on SessionStart:

{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "echo 'Reminder: use Bun, not npm. Run bun test before committing.'"
          }
        ]
      }
    ]
  }
}

Audit every command Claude runs#

{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.command' >> ~/.claude/command-log.txt"
          }
        ]
      }
    ]
  }
}

Verify tests pass before Claude stops#

A Stop hook that re-runs if tests are failing. Parse stop_hook_active to avoid infinite loops:

#!/bin/bash
INPUT=$(cat)
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0  # already ran once, let Claude stop
fi
if ! npm test --silent 2>/dev/null; then
  echo "Tests are failing — please fix them before finishing" >&2
  exit 2
fi
exit 0

Prompt-Based Hooks (type: "prompt")#

For decisions that require judgment, use a prompt hook instead of a shell command. Claude Code sends your prompt plus the event data to a Claude model (Haiku by default) and uses its response as the decision. The model must return JSON with "ok": true or "ok": false.

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Check if all requested tasks are complete. If anything remains unfinished, respond with {\"ok\": false, \"reason\": \"what still needs to be done\"}. Otherwise respond with {\"ok\": true}."
          }
        ]
      }
    ]
  }
}

When "ok": false, the "reason" is fed back to Claude so it keeps working. Use prompt hooks when the hook input data alone is enough to make a decision.

Agent-Based Hooks (type: "agent")#

When verification requires inspecting files or running commands, use an agent hook. A subagent is spawned with access to tools — it can read files, search code, and run commands before returning a decision. Agent hooks use the same "ok" / "reason" format as prompt hooks, with a 60-second default timeout and up to 50 tool-use turns.

{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "agent",
            "prompt": "Verify that all unit tests pass. Run the test suite and check the results. $ARGUMENTS",
            "timeout": 120
          }
        ]
      }
    ]
  }
}

Use prompt hooks when hook input data alone is enough to decide. Use agent hooks when you need to check actual codebase state (run tests, read files, etc.).

HTTP Hooks (type: "http")#

POST event data to an HTTP endpoint instead of running a shell command. Useful for shared audit services, cloud functions, or team logging:

{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "http",
            "url": "http://localhost:8080/hooks/tool-use",
            "headers": {
              "Authorization": "Bearer $MY_TOKEN"
            },
            "allowedEnvVars": ["MY_TOKEN"]
          }
        ]
      }
    ]
  }
}

The endpoint receives the same JSON a command hook would receive on stdin, and responds with the same JSON format to block or allow actions. HTTP status codes alone cannot block actions — use the response body. Header values support $VAR_NAME interpolation; only vars listed in allowedEnvVars are resolved.

Note: HTTP hooks must be added manually to settings JSON. The /hooks interactive menu only supports command hooks.

Troubleshooting#

Hook not firing

  • Run /hooks and confirm the hook appears under the correct event
  • Check the matcher is case-sensitive and matches the exact tool name
  • For PermissionRequest hooks: they don't fire in non-interactive (-p) mode — use PreToolUse instead

Hook error in output

Test your script manually:

echo '{"tool_name":"Bash","tool_input":{"command":"ls"}}' | ./my-hook.sh
echo $?   # check exit code
  • "command not found": use absolute paths or $CLAUDE_PROJECT_DIR
  • "jq: command not found": install jq (brew install jq / apt-get install jq)
  • Script not running at all: make it executable with chmod +x ./my-hook.sh

JSON validation failed

If your shell profile (~/.zshrc) has unconditional echo statements, the output prepends to your hook's JSON and breaks parsing. Fix by wrapping echos to only run interactively:

if [[ $- == *i* ]]; then
  echo "Shell ready"
fi

Stop hook loops forever

Parse stop_hook_active and exit 0 if it's true — this means the hook already ran once:

INPUT=$(cat)
if [ "$(echo "$INPUT" | jq -r '.stop_hook_active')" = "true" ]; then
  exit 0
fi

Debugging

Toggle verbose output with Ctrl+O to see hook output in the transcript. For full execution details run claude --debug.
